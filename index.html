SP-01 · SISR1 / SISR4
Migration Nationale vers Office 365 & Active Directory
SISR1SISR4B1.1B1.4

La Fondation regroupe plusieurs dizaines d'établissements médicaux répartis sur tout le territoire. Chaque site disposait d'une version d'Office différente (2013, 2016, 2019) et d'un annuaire AD cloisonné. L'objectif était d'harmoniser l'ensemble sur Microsoft 365 et de standardiser l'Active Directory.

Gain de temps
−60 à 70 %
sur le provisionnement des comptes
Périmètre
National
Tous les établissements
Criticité médicale : Une interruption même de quelques heures impacte les soignants. La migration a été planifiée par vagues, hors des heures de soins.

Phase 1 — Audit AD : Script PowerShell d'export complet des comptes AD pour détecter les comptes orphelins et inactifs avant migration.

POWERSHELL · Export_Audit_AD.ps1
# Audit complet des comptes AD avant migration M365
Get-ADUser -Filter * -Properties DisplayName,EmailAddress,Department,Enabled,LastLogonDate |
Select-Object DisplayName,SamAccountName,EmailAddress,Department,Enabled,LastLogonDate |
Export-Csv "C:\Audit\Export_AD_$(Get-Date -Format 'yyyyMMdd').csv" -Encoding UTF8 -NoTypeInformation
Write-Host "[OK] $((Get-ADUser -Filter *).Count) comptes exportes"

Phase 2 — Structure OU : Refonte de l'arborescence AD avec des UO normalisées par établissement et rôle (Soignants, Administratif, Stagiaires…).

Phase 3 — Azure AD Connect : Configuration de la synchronisation hybride on-premise ↔ Microsoft Entra ID, avec Password Hash Sync pour la continuité d'accès pendant la migration.

Phase 4 — Licences M365 : Attribution automatique des licences Business Premium via groupes dynamiques Azure AD selon le département.

// Pourquoi migrer vers M365 ?
Les licences Office sur site représentaient un coût de maintenance élevé. M365 apporte la mise à jour continue, Teams, SharePoint et la gestion centralisée via Intune — essentiels pour un réseau d'établissements multi-sites.
// Pourquoi standardiser l'Active Directory ?
Des AD hétérogènes rendaient impossible la gestion centralisée des droits. Un AD unifié permet d'appliquer des GPO cohérentes sur tous les sites et de provisionner un compte depuis un seul endroit.
// Pourquoi une migration par vagues ?
Dans un contexte médical, on ne peut pas se permettre une coupure globale. La migration par vagues (un établissement à la fois) limite le rayon d'impact et permet de valider chaque déploiement avant de passer au suivant.
Azure AD Connect / Entra ID — Synchronisation hybride on-premise ↔ cloud
Scripting PowerShell — Automatisation de l'audit et du provisionnement AD
Gestion de projet IT — Planification par vagues, communication avec les référents métier
Microsoft Purview — Étiquettes de sensibilité pour données médicales (à approfondir)
Prise de recul : Cette mission m'a appris l'importance de la phase d'audit avant toute migration. Un compte orphelin non détecté peut devenir un vecteur d'attaque. J'aurais dû anticiper davantage les dépendances entre applications métier et l'AD.

SP-02 · SISR2 / SISR3
Sécurité Réseau, VPN & Réponse à Incident de Compromission
SISR2SISR3B2.2B2.3
⚠ Incident réel : Un poste salarié a été compromis. WithSecure Elements a détecté un processus PowerShell encodé lancé depuis Outlook — signature d'une macro VBA malveillante reçue par spear phishing.
Temps de confinement
< 20 min
Données compromises
Aucune

Segmentation VLAN : Architecture réseau en 5 VLANs — Administration IT (10.10.10.0/24), Postes utilisateurs (10.10.20.0/24), Serveurs (10.10.30.0/24), IoT Médical (10.10.40.0/24), Invités (10.10.50.0/24).

VPN inter-sites : Tunnels IPsec/IKEv2 entre le siège et les établissements + VPN Always-On via Intune pour les postes nomades.

Réponse à l'incident : Isolation immédiate (port switch + quarantaine WithSecure), désactivation compte AD, analyse forensique des logs Windows.

POWERSHELL · Isolation_Compte_AD.ps1
# Désactivation immédiate du compte AD compromis
Disable-ADAccount -Identity "jdupont"
Write-EventLog -LogName "Security" -Source "Incident-SSI" `
    -EventId 9999 -Message "Compte isole - incident securite J+0"

# Réinitialisation forcée + obligation de changer le mot de passe
Set-ADAccountPassword -Identity "jdupont" `
    -NewPassword (ConvertTo-SecureString "TempP@ss2024!" -AsPlainText -Force) -Reset
Set-ADUser -Identity "jdupont" -ChangePasswordAtLogon $true
// Pourquoi segmenter en VLANs ?
Sans segmentation, un poste compromis peut atteindre tous les serveurs. Le VLAN IoT médical isole notamment les équipements de soins (pompes, moniteurs) qui ont souvent des firmwares non patchés et ne peuvent pas être mis à jour facilement.
// Pourquoi le VPN Always-On ?
Le personnel administratif travaille parfois en déplacement inter-sites. Sans VPN permanent, ils accèdent aux applications métier en clair sur des réseaux non maîtrisés. L'Always-On garantit que 100 % du trafic transite par notre pare-feu.
// Pourquoi notifier la CNIL même sans fuite ?
L'article 33 du RGPD impose une notification dans les 72h en cas de risque pour les droits des personnes. Même sans exfiltration confirmée, la notification préventive protège la Fondation juridiquement et démontre sa bonne foi.
Réponse à incident — Isolation, investigation forensique, remédiation complète
Framework MITRE ATT&CK — Lecture et interprétation des phases d'attaque
Obligations RGPD art. 33 — Notification CNIL, documentation de l'incident
Analyse de macro malveillante — Identification VBA / PowerShell encodé (à approfondir)
Prise de recul : J'ai compris que la rapidité prime sur la perfection lors d'un incident. Isoler en 5 minutes vaut mieux qu'analyser pendant 2 heures. La segmentation VLAN a été le facteur clé qui a empêché la propagation latérale vers les serveurs médicaux.

SP-03 · SISR1 / SISR5
Automatisation PowerShell & Support Utilisateurs N1/N2 via GLPI
SISR1SISR5B1.5B2.1

Les tâches récurrentes (création de comptes, montage de lecteurs réseau, inventaire matériel) représentaient plusieurs heures de travail manuel par semaine sur un parc multi-sites. L'objectif était d'automatiser 80 % de ces tâches et de structurer le support N1/N2 via GLPI.

Réduction tâches manuelles
−80 %
SLA incidents critiques
1 heure

Script inventaire automatique : Récupération via WMI des informations système (OS, RAM, CPU, dernier démarrage) de tous les postes du parc AD, exporté en CSV pour mise à jour de la CMDB GLPI.

POWERSHELL · Inventaire_Parc_SJDD.ps1
$computers = Get-ADComputer -Filter * -Properties *
$results   = @()
foreach ($pc in $computers) {
  try {
    $os  = Get-WmiObject Win32_OperatingSystem -ComputerName $pc.Name -EA Stop
    $ram = [math]::Round($os.TotalVisibleMemorySize / 1MB, 1)
    $results += [PSCustomObject]@{
      Nom=$pc.Name; OS=$os.Caption; RAM_Go=$ram; Statut="OK"
      Etablissement=($pc.DistinguishedName -split ',OU=')[2]
    }
  } catch { $results += [PSCustomObject]@{ Nom=$pc.Name; Statut="INACCESSIBLE" } }
}
$results | Export-Csv "Inventaire_$(Get-Date -f 'yyyyMMdd').csv" -Encoding UTF8

Montage lecteurs réseau (GPO Logon Script) : Attribution dynamique des lecteurs selon l'appartenance au groupe AD — déployé via GPO Logon Script sur tous les sites. Aucune configuration manuelle poste par poste.

// Pourquoi automatiser l'inventaire ?
Un inventaire manuel sur 30+ sites prend plusieurs jours. Le script le fait en moins d'une heure et produit un CSV directement importable dans GLPI. Sans inventaire à jour, on ne peut pas gérer le cycle de vie du matériel ni identifier les postes sous Windows 10 en fin de support.
// Pourquoi les lecteurs réseau via GPO ?
Sans script centralisé, chaque poste était configuré manuellement. Quand un utilisateur change de service, il faut reconfigurer son poste. Avec le GPO logon script, les lecteurs se montent automatiquement selon le groupe AD — zéro intervention manuelle.
// Pourquoi structurer le ticketing GLPI ?
Sans GLPI, les demandes arrivaient par email, téléphone, en direct. Impossible de mesurer les temps de résolution, de prioriser ou de constituer une base de connaissance. GLPI professionnalise le support et permet de justifier les ressources IT nécessaires.
PowerShell avancé — WMI, gestion d'erreurs try/catch, export CSV, boucles sur objets AD
GPO Logon Scripts — Déploiement et débogage de scripts au démarrage de session
Administration GLPI — Catégories, SLA, base de connaissance, CMDB
API GLPI REST — Création automatique de tickets depuis PowerShell (à implémenter)
Prise de recul : J'ai appris à penser en termes de scalabilité. Un script qui fonctionne sur 10 postes doit fonctionner sur 500 sans modification. La gestion des erreurs (try/catch) est indispensable en production pour éviter qu'un poste inaccessible stoppe tout le script.
SP-04 · SISR4 / SISR5
Déploiement Applicatif Netvie / Netfactu — Conformité M14/M22
SISR4SISR5B1.4B1.6

Netvie gère les dossiers résidents d'EHPAD (admissions, suivi médico-social). Netfactu gère la facturation des séjours et l'interface avec CEGI (comptabilité). Déploiement reproductible sur chaque établissement tout en préservant les paramètres locaux.

Criticité comptable : La facturation des séjours ne peut souffrir aucune interruption. Une erreur dans les exports vers CEGI peut bloquer la comptabilité d'un établissement entier. Conformité aux normes M14/M22 obligatoire.

Déploiement standardisé en 7 phases reproductibles : 1) Vérification prérequis (script PowerShell), 2) Snapshot VM pré-installation, 3) Installation + création base SQL, 4) Paramétrage codes analytiques, 5) Intégration CEGI + tests exports, 6) Recette fonctionnelle avec le comptable référent, 7) Formation utilisateurs + go-live avec surveillance 48h.

ComptableSaisie + validationSaisie + édition factures

// Pourquoi un déploiement standardisé ?
30 établissements, c'est potentiellement 30 déploiements. Sans procédure reproductible, chaque installation devient un projet en soi avec des risques d'incohérence. La standardisation réduit le temps de déploiement et garantit un niveau de service homogène sur tous les sites.
// Pourquoi la conformité M14/M22 ?
Les établissements médico-sociaux publics sont soumis à des instructions comptables spécifiques. Les exports vers CEGI doivent respecter ces normes pour que la comptabilité soit légalement recevable lors des contrôles de l'ARS ou de la Cour des comptes.
// Pourquoi impliquer les comptables dans la recette ?
Un déploiement validé techniquement peut être fonctionnellement incorrect. Seul le comptable peut vérifier que les codes analytiques et les exports sont conformes à leur réalité métier. Sans recette fonctionnelle, on livre un outil qui ne répond pas au besoin réel.
Administration SQL Server — Création de bases, ODBC, sauvegardes, SQL Agent
Gestion de projet IT — Phases de déploiement, recette fonctionnelle, go-live
Relation avec les métiers — Traduction des besoins comptables en configuration technique
Rédaction de procédures — Documentation de déploiement réutilisable (en cours de formalisation)
Prise de recul : L'aspect le plus complexe n'était pas l'installation du logiciel, mais de comprendre les enjeux comptables pour paramétrer correctement les exports. Cette mission m'a confirmé que la communication avec les utilisateurs finaux est une compétence technique à part entière.

2024 · T1
Microsoft Entra ID remplace Azure AD
Renommage et restructuration d'Azure Active Directory en Microsoft Entra ID. Impact direct sur la gestion des identités hybrides et les connecteurs AD Connect utilisés en entreprise.
Source : Microsoft Learn / Microsoft Blog
2024 · T2
ANSSI — Guide sécurisation Active Directory
Mise à jour des recommandations ANSSI pour la sécurisation des AD. Points clés : modèle Tiering, suppression des droits admin permanents, surveillance des opérations DCSync.
Source : ANSSI.fr
2024 · T2
Ransomwares ciblant le secteur médical
Le CERT-FR alerte sur une recrudescence des attaques ransomware contre les établissements de santé français. LockBit 3.0 et BlackCat/ALPHV parmi les groupes les plus actifs en 2024.
Source : CERT-FR / ANSSI
2024 · T3
Windows Server 2025 — Nouveautés SMB
WS2025 apporte SMB over QUIC (accès aux partages sans VPN), SMB signing obligatoire par défaut, et des améliorations pour les environnements hybrides.
Source : Microsoft Docs
2024 · T4
NIS2 — Nouvelle directive cybersécurité EU
La directive NIS2 étend les obligations de cybersécurité aux établissements de santé. Notification des incidents en 24h, tests de résilience annuels, responsabilité pénale des dirigeants.
Source : ENISA / Journal Officiel UE
2025 · T1
Microsoft Intune — Gestion unifiée endpoints
Intune se renforce comme solution MDM/MAM unifiée. Déploiement d'applications, conformité des postes et gestion des politiques sans serveur SCCM on-premise — directement applicable à la Fondation.
Source : Microsoft Tech Community
